## 思考プロセスのメモ

### enabledPokemonの設計について
当初「インポートでマスタ自体を差し替える」案(B)も検討したが、「全ポケモンマスタ + 有効リスト切り替え」案(A)を採用。
理由: 一度登録した有利不利データはpokemonIdをキーにしているため、マスタが変わっても安全に引き継げる。
マスタを差し替える案だと、一時的に除外されたポケモンの有利不利が参照できなくなるリスクがある。

### リージョンフォームのID体系
- PokeAPIの species エンドポイント: 1〜1025の通常ID
- PokeAPIの pokemon エンドポイント: 10000番台がフォームバリアント（10008〜10252など）
- data.js でこの高IDをそのまま使用。store.js の DEFAULT_ENABLED_IDS にも含む。

### カバレッジ判定の閾値設計
「上位50匹全部2匹カバー」は現実的に難しい → 上位20/30匹だけ2匹カバーを求め、残りは1匹カバーでOKに。
これによってユーザーが「どこまで厳しくするか」を試合ごとに調整できる。

### CORSの回避方針
Champions公式サイトへの直接フェッチはCORS制限で不可。
解決策:
- A. プロキシサーバー経由（維持コストが高い）
- B. ブラウザconsoleで手動抽出 → JSONファイルとして保存 → 設定画面でインポート

Bを採用。抽出コードと結果JSONをリポジトリに含め、次回更新時の参考にできるようにした。

### Firebase IndexedDB エラー修正（firebase.js）

**原因:**
Firestore の `enablePersistence` によりすべての書き込みが IndexedDB 経由になる。
ブラウザが IndexedDB 接続を切断したとき（長時間放置・メモリ不足・複数タブ等）に書き込みがブロックされる。

**対処:**
- `enablePersistence` のエラーをコード別に分類して適切なログを出すよう改善（`failed-precondition` = 複数タブ、`unimplemented` = 非対応ブラウザ）
- `pushToCloud` で `IndexedDb` を含むエラーを検出したらページリロードダイアログを表示
- 即時対処はページ再読み込み（データは失われない）

**コミット:** `1a197fd` Fix IndexedDB sync error with better error handling and reload prompt
